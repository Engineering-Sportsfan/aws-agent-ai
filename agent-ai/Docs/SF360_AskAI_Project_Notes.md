# SportsFan360 — AskAI Feature · Project Handover

---

## 1. Project Overview

**SportsFan360** is an AI-powered cricket analytics web app. The **AskSportsFan360** feature is a conversational AI assistant embedded in the app that answers cricket-related questions for users — career stats, season comparisons, team records, current standings, and so on.

The assistant is built around a tool-calling pattern: a language model interprets the user's question and routes it to one of several data-retrieval tools. The numerical answer is then formatted in Python and returned to the user. This separation keeps factual answers grounded in data rather than generated by the model, which is the core design principle of the feature.

## 3. Local Development Setup

Project root: `sportsfan360-ai-agent-main/`

**Main files:**
- `agent.py` — FastAPI backend entry point; defines the API endpoints and the tool-calling loop
- `data_loader.py` — data access layer; reads from local data files and exposes query functions
- `query_local.py` — local query tool for verifying behaviour without going through the API

**Setup:**

```bash
conda create -n sf360-askai python=3.11
conda activate sf360-askai
pip install -r requirements.txt
```

**Required environment variables** (see `.env.example` in the repo root):
- `GROQ_API_KEY` — for the LLM that handles question routing

**Run locally:**

```bash
python -m uvicorn agent:app
```

Server defaults to `http://localhost:8000`. Endpoints can be tested directly or via `query_local.py`.

> Note: avoid running uvicorn with `--reload`. The auto-reload behaviour has previously caused runaway LLM API calls during development. Run without flags.

## 4. Architecture

```
User question
     ↓
Groq LLM (llama-3.3-70b) — picks the right tool to answer this question
     ↓
Tool executor → data layer → returns structured result
     ↓
Python formats the answer (no LLM round-trip for numerical content)
     ↓
JSON response: { answer, chart_title, chart_data }
```

**Key design decisions:**

- **Two-track answering.** Factual / numerical questions go through the data tools; pure knowledge or trivia questions (e.g. "who invented cricket?") are answered by the LLM directly. The router decides which path a given question takes.
- **The LLM never sees the numbers when composing the final answer.** This avoids the most common failure mode of LLM-based stat lookups, where the model invents plausible-looking but wrong numbers. Python formats every data-driven answer.
- **Chart payload alongside text.** The response always includes `chart_title` and `chart_data` fields. The frontend uses these to render an inline chart when relevant, or ignore them when not.

## 5. Data Source

The underlying cricket data is derived from **Cricsheet** (https://cricsheet.org), a public open-source archive of structured cricket match data. The current dataset in the project is a **static snapshot** prepared for the AskAI feature and committed to the repo.

The snapshot is not refreshed automatically. If the team wants to incorporate newer match data in future, a Cricsheet ingestion pipeline will need to be set up separately — this would be a standalone task, independent of the AskAI codebase itself. The data layer (`data_loader.py`) is the integration point; any new pipeline just needs to produce files matching the existing format.

## 6. Next Steps — Integration with the Main App

The front-end team has shared an integration document (`SF360_AskAI_Integration.md`) proposing the migration from the current iframe embed to a native React component inside the main Next.js app.

**Proposed scope:**

1. **Replace the iframe** with a native React component inside the parent Next.js app, so the chat UI sits in the same DOM as the rest of the app
2. **Add `POST /chat` endpoint** to the Python backend, alongside the existing `GET /ask?question=` (which can stay during the transition)
3. **Firebase for conversation history** — replace the current in-memory context with persistent storage per user
4. **User personalisation** — inject the user's profile data (followed teams, favourite players) from Firebase into queries so answers can be tailored
5. **Streaming responses** — add SSE (Server-Sent Events) for word-by-word response display, which makes the chat feel more responsive

**Python backend changes needed:**

- New `POST /chat` endpoint accepting `{query, conversation_history, user_id, session_id}`
- Continue passing `chart_title` and `chart_data` in the response alongside `answer`
- The existing routing and tool-execution logic stays unchanged — this is purely an interface addition

The existing `GET /ask` endpoint should remain operational until the Next.js integration is fully cut over, to avoid breaking the current iframe.

## 7. Operational Notes

**Deployment:**
- Render auto-deploys from the `main` branch of the GitHub repo on every push
- Build / deploy logs are visible in the Render dashboard
- The Vercel frontend deploys independently and is also tied to the GitHub repo

**Monitoring:**
- `cron-job.org` is set up to ping the `/health` endpoint every 10 minutes (Render free tier sleeps after inactivity, so this keeps it warm)
- API request and error logs are visible in the Render dashboard

**Things to be aware of:**
- The GROQ free tier has a daily token cap. For production scale, the account will eventually need to be upgraded.
- Some queries gracefully return empty results if the underlying data doesn't cover them — this is expected behaviour, not an error state.
- When changes are pushed to the repo, give Render a minute or two to redeploy before testing changes against the live URL.
